The maths behind each landscape's unique sequence of seed numbers

There are 10,000 different landscapes in The Sentinel, numbered from 0000 to 9999. That's a huge number, so huge that Geoff Crammond said that he never thought anyone would go so far as to finish the game (see Wikipedia).

The key to this huge number of landscapes is a pseudo-random number generator that can be seeded with the landscape number to generate the same, predictable sequence of numbers for each specific landscape. This approach was used in games like Elite to spectacular effect - see the Elite deep dive on galaxy and system seeds for example - but The Sentinel takes its random number generation to the next level, using them to generate not only the landscapes, but the secret codes too:

For more information on how the game generates landscapes from sequences of pseudo-random numbers, see the deep dive on generating the landscape. For this article, we're going to take a look at how those numbers are generated in the first place.

Linear feedback shift register
------------------------------

The numbers that power The Sentinel's 10,000 landscapes are generated by two very short routines. The InitialiseSeeds routine sets up the pseudo-random number generator and GetNextSeedNumber fetches the next number from the pseudo-random number generator. Because these numbers are used to create landscapes, I've called them "seeds", so from now on I'll talk about the seed generator routines and how they generate sequences of seeds for each landscape.

For anyone who's looked into procedural generation on old computers, the maths behind the landscape seed generation will be a familiar one. The Sentinel uses a linear feedback shift register (LFSR) that consists of five bytes at seedNumberLFSR(4 3 2 1 0). The register shifts bits to the left, from low to high, and each shift generates one bit of the next seed, so to generate a new random number, we shift the register eight times.

The key to the pseudo-random nature of this shifting is the bit that we rotate into bit 0 during the shift. The value that we move into bit 0 is this:

  Bit 3 of seedNumberLFSR+2 EOR bit 0 of seedNumberLFSR+4

which is the same as:

  Bit 19 of seedNumberLFSR EOR bit 32 of seedNumberLFSR

Here's a diagram showing how the feedback loop works:

    LFSR+4   LFSR+3   LFSR+2   LFSR+1   LFSR+0

        32       24       16        8        0
         |        |        |        |        |
         v        v        v        v        v
  76543210 76543210 76543210 76543210 76543210  <-----+
         ^              ^                             |
         |              |                             |
         \____ EOR ____/                              |
                |                                     |
                +------------------------------------ +

The GetNextSeedNumber routine therefore extracts bits 19 and 32 from the shift register in seedNumberLFSR(4 3 2 1 0) and EORs them before rotating the register to the left, inserting the result of the EOR into bit 0. This process ensures that the numbers that are generated are random enough to create 10,000 landscapes.

The correct term for a bit that we extract and process in order to feedback into a shift register is a "tap", so in the above we are "tapping" the linear feedback shift register in two places. Slightly confusingly, when we talk about tapping, we actually refer to the locations of the taps after the shift register has shifted, rather than before, so although we are extracting bits 19 and 32 from the register before shifting, the correct terminology for this is a "33-bit linear feedback shift register with taps at bits 20 and 33".

If this sounds familiar, then you may have been reading my analysis of Lander, the world's first game for the ARM processor. Lander uses exactly the same system to produce random numbers, and although it doesn't use them to create its famous tiled landscapes, it does use code from the original ARM Evaluation System (see the Lander deep dive on random numbers for details).

So The Sentinel generates seeds using the exact same algorithm as the original ARM development team and the very first ARM game... but The Sentinel does it in just eight bits.

Initialising the shift register
-------------------------------

The beauty of the linear feedback shift register is that if you initialise the first two bytes of the register with specific values, then the output of the register is predictable and repeatable. The Sentinel uses this fact to set up the generator, ready for generating the landscape.

Specifically, it takes the landscape number in binary coded decimal (BCD) and sets them as the first two bytes. The landscape number ranges from 0000 to 9999, and as BCD matches the way that humans write down numbers, this range of landscape numbers matches the range of a 16-bit BCD number. For example, landscape 1234 would be represented in BCD using a high byte of &12 and a low byte of &34, so to initialise the seed generator for this landscape, we set seedNumberLFSR+1 to the high byte (&12) and seedNumberLFSR to the low byte (&34).

There is one more step, and that is done in the ResetTilesObjects routine. Apart from the first two bytes of the register, which contain the landscape number, the other bytes are all zeroed, and then bit 16 is set. This last step is vital as otherwise landscape 0000 would start with a register entirely made up of zeroes, and that could only ever produce more zeroes, as 0 EOR 0 is 0.
