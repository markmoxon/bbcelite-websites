Creating, deleting and managing up to 64 objects on the landscape

The tiled landscape of The Sentinel can be home to up to a number of 3D objects, from trees and boulders to sentries and meanies. Up to 64 of these objects can be created ("spawned"), deleted, placed on tiles and removed from tiles, and some objects can even be stacked on top of one another, like this robot on top of a boulder:

In this article we look at how objects are managed. See the deep dive on stacking objects for more information about object stacks, and check out the deep dives on 3D object definitions and drawing objects for details of how all these objects are drawn on-screen.

Object management
-----------------

If we want to add an object to the landscape, then the first thing we need to do is call GetObjectNumber to fetch an object number that we can use for the new object. We pass the object type for the new object in A, as follows:

  • 0 = Robot (the player object is one of these)
  • 1 = Sentry
  • 2 = Tree
  • 3 = Boulder
  • 4 = Meanie
  • 5 = The Sentinel
  • 6 = The Sentinel's tower
  • 7 = 3D text block: no block (left), block (right)
  • 8 = 3D text block: Block (left), No block (right)
  • 9 = 3D text block: Block (left), block (right)

The last three objects are only used when drawing the large 3D text on the title screens - they aren't used in-game at all, so we can ignore these for the rest of this article. See the deep dive on drawing 3D text using blocks for more information.

When we create an object, it has an object number allocated to it. This object number is in the range 0 to 63, as the system can support up to 64 objects. The data for each object is stored in multiple places, but one of the most important is the objectFlags table. If the entry in the X-th byte in objectFlags has bit 7 set, this indicates that object number #X has not yet been allocated to an object; conversely, if bit 7 is clear, then object number #X refers to an object that we have spawned. That object might not appear on the landscape, but the object number has been allocated.

When the game is first started, all 64 entries in the objectFlags table are set to contain %10000000, to indicate that no object numbers have been allocated. The table is also reset by the ResetVariables and ResetTilesObjects routines.

The GetObjectNumber routine therefore looks for an object number that isn't currently allocated to an object so we can use it to spawn a new object via one of the spawning routines. It does this by searching the objectFlags table for a free number to allocate - i.e. an entry with bit 7 set. It starts at the entry for object #63 and works its way down the table until it finds a free number.

If it finds a new number then it stores the object type in the X-th entry in the objectTypes table and returns to the caller with the new object's number and the flags set on success or failure (for example, if we already have 64 objects and can't add any more).

The next step is to spawn the object using this object number and add the object to the landscape. This is done by the SpawnObjectOnTile routine, which starts by checking whether the tile is a suitable home for the new object (i.e. whether the tile is empty, or contains an object stack that we can add the new object to). If we can place the object on the tile, then SpawnObjectOnTile goes ahead and creates the object as follows (assuming we are spawning object number #X on the tile):

  • The X-th entry in (xObject, yObject, zObject) is set to the 3D coordinate of the newly added object, where the y-coordinate is a 16-bit value in yObject(Hi Lo) that contains an integer and fractional part.
  • The X-th entry in objectFlags, bit 7 is clear to indicate that object number #X has now been allocated to an object.
  • The X-th entry in objectFlags has bit 6 set if we add the object on top of a boulder or tower (and the object number of the boulder/tower is in bits 0-5).
  • The X-th entry in yObjectLo is set to 224, so objects are spawned at a height of 224 above the tile itself, or 0.875 tile widths (this is used for the gaze vector calculations, so this is effectively the height of the object's eyes above the tile).
  • The X-th entry in objectPitchAngle is set to -11, so objects look slightly downwards at a pitch angle of 15.5 degrees.
  • The X-th entry in objectYawAngle is set to a multiple of 11.25 degrees, as determined by the next seed, with 96 added to it (135 degrees). This is presumably intended to make the player's rotation work well on the starting level, though it is added to all objects and not just the player object (though it is also overridden after spawning for some objects).
  • The tileData for the tile is set to the object number in X in bits 0 to 5, and bits 6 and 7 are set to indicate that the tile contains an object.

GetObjectCoords This routine fetches the Cartesian coordinates of object #X as three 24-bit numbers, as follows: xCoord(Hi Lo Bot) yCoord(Hi Lo Bot) zCoord(Hi Lo Bot) The high byte can be thought of as the integer with low and bottom bytes as the fractional part. Tile corners are integers in the x- and z-axis, and objects are placed in the centre of tiles by setting the fractional part to 0.5. So an object that's been placed directly on the tile that's anchored at (2, y, 3) will appear at (2.5, y, 3.5). The y-coordinate - i.e. the altitude of the object above the tile itself - depends on the object type and whether it is stacked on top of another object. From another dive: This also fetches the next seed number for the landscape and uses this to set the object's yaw angle (see the deep dive on object management for details of this process).

Object variables and routines
-----------------------------

There are quite a few tables that are involved in managing objects in the game world. Here's a list that we'll explain in more detail below.

VariableDescription
objectTypesA table of object types for up to 64 in-game objects
objectFlagsObject flags for up to 64 objects (object number allocation, stack details)
objectPitchAngleThe pitch angle for each object (i.e. the vertical direction in which they are facing)
objectYawAngleThe yaw angle for each object (i.e. the horizontal direction inwhich they are facing)
xObjectThe integer x-coordinate of the tile containing the object (i.e. along the left-right axis)
yObjectHiThe integer part of the object's y-coordinate, i.e. the altitude in tile blocks
yObjectLoThe fractional part of the object's y-coordinate, i.e. the fractional altitude within the tile block
zObjectThe integer z-coordinate of the tile containing the object (i.e. along the axis into and out of the screen)
objectTypeEnergyThe amount of energy encapsulated in each type of object, i.e. the amount of energy acquired or spent when absorbing or creating the object
objectHalfWidthEach object's width in terms of tile widths, for half of the object only

There are also a few routines that enable us to manage objects in the landscape:

SubroutineDescription
GetObjectNumberFetch an object number that we can use for a new object
DeleteObjectDelete an object, removing it from the landscape and vacating its object number
SpawnObjectOnTileSpawn an object on a tile, putting it on top of any existing boulders or towers
SpawnObjectBelowAttempt to spawn an object on a tile that is below the maximum altitude specified in A
GetObjectCoordsGet an object's coordinates
